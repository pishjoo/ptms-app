**Release Date:** 2026-07-14

## 1. Scope

This document freezes the architecture of PTMS as of Sprint 12. It captures the complete structural, strategic, and tactical design decisions that form the foundation for future development cycles. No new features are introduced — this is a pure audit and freeze.

---

## 2. Architecture Overview

PTMS follows a **layered domain-driven architecture** with the following layers:

| Layer | Directory | Responsibility |
|---|---|---|
| **Presentation** | `app/`, `components/` | Next.js App Router pages, React components, UI rendering |
| **Application** | `server/application/` | Orchestration services (ImportPreviewService, TradeCaseAnalysisService, TradeImportOrchestrator) |
| **Domain** | `server/domain/` | Business logic, entities, rules, calculators, state engine, value objects |
| **Import Pipeline** | `server/import/` | File detection, Excel reading, mapping, change detection, merge engine |
| **Infrastructure** | `prisma/` | Database schema (Prisma ORM) |
| **Mock** | `server/mock/` | Test doubles and mock providers |

### Key Design Choices

1. **Next.js App Router** — Full-stack framework with server components and API routes
2. **Prisma ORM** — Database access with schema-first approach
3. **Dependency Injection** — Constructor-based DI throughout all services (no DI container)
4. **Immutable Types** — All interfaces use `readonly` modifiers
5. **Pure Functions** — State engine uses exported pure functions for stage/action/deadline logic
6. **TypeScript strict mode** — Enabled with `strict: true`

## 3. Layer Architecture Scorecard

### Clean Architecture Compliance: 7/10

**Strengths:**
- Domain layer has zero dependencies on frameworks or infrastructure
- Application services orchestrate domain logic without embedding it
- Import pipeline cleanly separates detection → mapping → change detection → preview
- All domain entities are plain TypeScript interfaces/classes with no Prisma coupling

**Weaknesses:**
- `TradeImportOrchestrator` mixes application orchestration with import pipeline concerns — it lives in `server/application/` but contains Excel-specific logic (sheet-to-json conversion, worksheet loading) that violates the Dependency Inversion Principle
- `TradeCaseAnalyzer` in `server/domain/services/` imports from `server/domain/rules/` which is acceptable, but the `calculateValue()` method with `as TValue` cast and `0 as TValue` fallback is fragile
- There is no explicit **Repository** or **Gateway** interface — the "ExistingRecordsProvider" in the orchestrator is a raw function type rather than a proper repository abstraction
- The `server/domain/trade-case/business/` package overlaps with the newer `server/domain/trade-case/state/` package — both contain `trade-case-health.ts`, `trade-case-progress.ts` with different interfaces

### DDD Compliance: 7/10

**Strengths:**
- `TradeCaseAggregate` is a well-defined aggregate root
- Value objects are used throughout (ImportWarning, ImportError, PreviewChange, etc.)
- Domain services are stateless and operate on aggregates
- Ubiquitous language is consistent (TradeCase, Registration, Allocation, Origin, Declaration, Commitment)

**Weaknesses:**
- No explicit bounded context boundaries — the entire `server/domain/` is a single context
- `server/domain/trade-case/dto/` mixes DTOs with domain — DTOs should be in the application layer
- `TradeCaseSummary` in the domain DTO package is used by both domain and application, blurring the boundary
- No domain events or event-driven communication between aggregates

### SOLID Compliance: 7.5/10

| Principle | Score | Notes |
|---|---|---|
| **S**ingle Responsibility | 8 | Most classes have clear responsibilities. `TradeImportOrchestrator` is the main exception. |
| **O**pen/Closed | 7 | Rules engine uses `TradeCaseRule` interface — good. State engine uses pure functions — extensible. But calculators use a fragile ID-based lookup. |
| **L**iskov Substitution | 8 | All interfaces are well-defined. No inheritance abuse. |
| **I**nterface Segregation | 7 | Some interfaces (like `TradeCaseAggregate`) are large, but appropriate for an aggregate root. `ChangeSet` is well-segregated. |
| **D**ependency Inversion | 7 | Constructor-based DI used throughout. But `TradeImportOrchestrator` creates its own dependencies with `new` instead of receiving them. |

## 4. Import Pipeline Analysis

**Strengths:**
- Clean separation: Detection → Loading → Validation → Mapping → Change Detection → Preview
- `ChangeDetector` is generic and reusable
- `MergeEngine` supports manual field policies
- DTOs are well-typed with readonly interfaces

**Weaknesses:**
- `TradeImportOrchestrator.previewImport()` is 160+ lines with inline Excel manipulation — SRP violation
- The `!rows` attachment (`(sheet as any)['!rows'] = rows`) is a type-unsafe hack
- `tryMap` closure captures state in outer scope — impure, hard to test
- `normalizeKey()` magic string logic creates implicit coupling between logical names and worksheet lookups
- `existingRecordsProvider` is typed as `(profileType: string) => readonly unknown[]` — too generic, loses type safety

---

## 5. Domain Model Analysis

**Strengths:**
- TradeCaseAggregate cleanly models the five-stage lifecycle
- Rules engine (`TradeCaseRule` interface) is extensible via `RuleRegistry`
- State engine (`TradeCaseStateEngine`) provides a single-entry `evaluate()` method
- All outcomes are typed interfaces with `readonly` modifiers

**Weaknesses:**
- **Duplication**: `server/domain/trade-case/business/trade-case-health.ts` and `server/domain/trade-case/business/trade-case-progress.ts` overlap with `server/domain/trade-case/state/` counterparts. The old `business/` services compute similar but subtly different results (e.g., `TradeCaseHealth.healthStatus: 'ON_TRACK' | 'AT_RISK' | 'DELAYED' | 'COMPLETED'` vs new `TradeCaseHealth.status: 'HEALTHY' | 'NEEDS_ATTENTION' | 'CRITICAL'`). Both cannot coexist long-term.
- All six rule implementations are stubs — they return `{ passed: true, message: 'Rule not implemented yet.' }`
- `TradeCaseBalanceService` is a pass-through that just copies fields from the aggregate — questionable value
- `TradeCaseStatusService` and `TradeCaseProgressService` in `business/` duplicate logic that now exists in `state/`

## 6. Cross-Cutting Concerns

### Security: 6/10
- No authentication middleware visible
- No role-based access control implemented in the server layer
- No input validation at the API boundary (no Zod schemas, no DTO validation)
- TypeScript strict mode provides some compile-time safety

### Performance: 7/10
- Immutability via Object.freeze and spread operators may create GC pressure for large imports
- No streaming or pagination for large import previews
- Excel reading happens synchronously — no streaming parser
- No observable caching strategy

### Testability: 8/10
- Constructor-based DI makes all services mockable
- Pure functions in state engine are trivially testable
- All interfaces are defined, enabling easy test doubles
- **Missing:** No test files exist in the repository — zero unit, integration, or E2E tests

### Extensibility: 7/10
- Rules engine is extensible via `TradeCaseRule` interface
- State engine functions are composable
- Import pipeline stages are replaceable
- No plugin/module system for adding new trade case stages or import sources
